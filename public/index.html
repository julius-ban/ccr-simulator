ES CCR Failover / Failback 자동화 콘솔

주센터 · DR센터 클러스터 등록부터 CCR 연동, Failover/Failback 실행까지

🧪 시뮬레이션 모드가 켜져 있습니다 — CCR 연동/모니터링/Failover/Failback 버튼은 실제 클러스터를 건드리지 않고 가짜 데이터로 동작합니다.

📖 용어집

1. 클러스터 등록

2. 샘플 벡터 인덱스 생성 (rag-vectors)

기본값은 제공해주신 rag-vectors 스펙(5샤드, HNSW, content_analyzer 등)을 그대로 반영했습니다.

이 인덱스는 반드시 아래 3번에서 "Leader 클러스터(주센터)"로 지정할 클러스터에 생성해야 합니다. CCR follower는 자기가 알아서 데이터를 찾는 게 아니라, Leader 클러스터 + Leader 인덱스 이름을 그대로 지정받아서 그 인덱스를 원본 삼아 복제합니다. 즉 여기서 만든 인덱스 이름을 3번의 "Leader 인덱스" 칸에 동일하게 입력해야 follow가 성공합니다 (기본값은 둘 다 rag-vectors로 이미 맞춰져 있습니다).

고급: settings/mappings JSON 직접 입력 (비워두면 기본 rag-vectors 스펙 사용)
  • PUT /{인덱스} — 벡터(임베딩) 필드를 포함한 인덱스를 생성합니다. 매핑/설정은 주신 rag-vectors 스펙을 그대로 사용합니다.
  • POST /_bulk — 테스트용 문서를 배치 단위로 대량 삽입합니다. 한 번에 다 보내지 않고 배치별로 나눠 보내서 서버 부하를 조절합니다.

3. CCR 연동 (정방향)

"Leader 클러스터"와 "Leader 인덱스"는 위 2번에서 실제로 인덱스를 만든 클러스터/이름과 반드시 일치해야 합니다.

  • POST /_security/cross_cluster/api_key — 주센터에 "DR센터가 이 인덱스를 검색/복제할 수 있는" 권한만 가진 전용 키를 발급합니다. 이 키가 없으면 DR센터는 주센터에 접근할 수 없습니다. 
    {
  "name": "ccr-<remote_alias>-key",
  "access": {
    "search": [ { "names": ["<leader_index>"] } ],
    "replication": [ { "names": ["<leader_index>"] } ]
  }
}
  • (수동) elasticsearch-keystore add — 발급된 키를 DR센터 각 노드의 비밀 저장소에 등록합니다. OS 명령이라 REST로는 못 하고 직접 실행해야 합니다.
  • PUT /_cluster/settings — DR센터에게 "주센터가 어디 있는지" 주소를 알려줍니다 (원격 클러스터 등록). 
    {
  "persistent": {
    "cluster": {
      "remote": {
        "<remote_alias>": {
          "mode": "proxy",
          "proxy_address": "<leader_host>:<leader_proxy_port>",
          "server_name": "<leader_host>"
        }
      }
    }
  }
}
  • PUT /{인덱스}/_ccr/follow — 실제 복제를 시작합니다. 이 순간부터 주센터의 데이터가 DR센터로 실시간 미러링됩니다. 
    {
  "remote_cluster": "<remote_alias>",
  "leader_index": "<leader_index>"
}

4. CCR 복제 모니터링

  • GET /{인덱스}/_ccr/stats — 복제 지연(lag), 지금까지 복제된 문서 수, 에러 여부 등을 5초 간격으로 조회합니다. 실제 데이터가 오가지는 않고 상태만 확인합니다.
복제 지연(lag) 0

5. Failover / Failback 실행

    

      Failover — 주센터에 장애가 생겼을 때, DR센터를 독립적으로 쓰기 가능한 상태로 전환해서 서비스를 계속 이어가는 것

      Failback — 장애가 복구된 뒤, DR센터에 쌓인 데이터를 다시 주센터로 되돌리고 원래처럼 주센터 중심 운영으로 복귀하는 것
    

    

      
      
    

    

      
    

    

      
    
        
  • POST /{인덱스}/_ccr/pause_follow — 복제를 일시정지합니다 (더는 주센터 데이터를 받지 않음).
    • 
        
  • POST /{인덱스}/_close — 인덱스를 잠깐 닫아서 설정을 안전하게 바꿀 수 있는 상태로 만듭니다.
    • 
        
  • POST /{인덱스}/_ccr/unfollow — 팔로워 관계를 완전히 해제합니다. 이제 이 인덱스는 독립적인, 쓰기가 가능한 일반 인덱스가 됩니다.
    • 
        
  • POST /{인덱스}/_open — 인덱스를 다시 열어서 애플리케이션이 바로 쓰기/읽기를 할 수 있게 합니다.
    • 
      

    


    

    

      
      
      
    

    

      
      
      
      
      
      
    

    

      
    
        
  • DELETE /{인덱스} — 원래 주센터에 남아있던 예전 인덱스를 지웁니다. DR센터로부터 다시 데이터를 받을 자리를 비우는 것입니다.
    • 
        
  • POST /_security/cross_cluster/api_key — 이번엔 반대로, 주센터가 DR센터 데이터를 가져올 수 있는 키를 DR센터에서 발급합니다.
    • 
        
  • PUT /_cluster/settings + PUT /_ccr/follow — 주센터를 DR센터의 팔로워로 등록해서, Failover 중 DR에 쌓인 데이터를 다시 주센터로 복제해옵니다 (역방향 CCR).
    • 
        
  • pause_follow → close → unfollow → open (⑥ 버튼) — 문서 수가 양쪽 일치하는 걸 확인한 뒤, 역방향 연결을 끊고 주센터를 다시 독립적인 쓰기 가능 상태로 되돌립니다. 이후 정방향 CCR을 다시 구성하면 원래 운영 형태로 복귀합니다.
    •
