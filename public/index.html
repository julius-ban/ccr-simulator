2. 샘플 벡터 인덱스 생성 (rag-vectors)

    
기본값은 제공해주신 rag-vectors 스펙(5샤드, HNSW, content_analyzer 등)을 그대로 반영했습니다.

    

      이 인덱스는 반드시 아래 3번에서 "Leader 클러스터(주센터)"로 지정할 클러스터에 생성해야 합니다.
      CCR follower는 자기가 알아서 데이터를 찾는 게 아니라, Leader 클러스터 + Leader 인덱스 이름을
      그대로 지정받아서 그 인덱스를 원본 삼아 복제합니다. 즉 여기서 만든 인덱스 이름을 3번의 "Leader 인덱스" 칸에
      동일하게 입력해야 follow가 성공합니다 (기본값은 둘 다 rag-vectors로 이미 맞춰져 있습니다).
    

    

      
      
      
      
      
      
      
    

    

      고급: settings/mappings JSON 직접 입력 (비워두면 기본 rag-vectors 스펙 사용)
      
    

    

      
      
    

  


  
  

    
3. CCR 연동 (정방향)

    
"Leader 클러스터"와 "Leader 인덱스"는 위 2번에서 실제로 인덱스를 만든 클러스터/이름과 반드시 일치해야 합니다.

    

      
      
      
      
      
      
      
      
    


    

      
    

    


    

      TLS 인증서 기반 인증은 API Key 발급 없이, 양쪽 클러스터의 transport 계층(포트 9300)이
      같은 CA를 신뢰하도록 인증서를 미리 맞춰둬야 합니다 (elasticsearch-certutil로 CA/인증서를
      발급해서 각 노드에 배포 — REST로는 자동화할 수 없는 수동 작업입니다). 준비되어 있다는 전제로
      "③ Remote Cluster 등록"부터 바로 진행하시면 됩니다.
    


    

      
      
    

    

      
      
    

    

    


    

    
다중 인덱스 일괄 Follow

    
이미 존재하는 인덱스 여러 개를 한 번에 follow합니다. 위 ③번 Remote 등록이 먼저 되어 있어야 합니다.

    

      
      
    

    

      
    

    


    

    
Auto-follow 패턴 구성 (신규 인덱스 자동 복제)

    
위 ③번에서 등록한 remote alias를 그대로 씁니다. 리더에 이 패턴에 맞는 인덱스가
      새로 생성될 때마다 자동으로 팔로워가 만들어집니다. 지금 이미 존재하는 인덱스는
      대상이 아닙니다 (기존 인덱스는 위 ④번 또는 "다중 인덱스 일괄 Follow"로 개별/일괄 follow 하세요).

5. Failover / Failback 실행

    

      Failover — 주센터에 장애가 생겼을 때, DR센터를 독립적으로 쓰기 가능한 상태로 전환해서 서비스를 계속 이어가는 것

      Failback — 장애가 복구된 뒤, DR센터에 쌓인 데이터를 다시 주센터로 되돌리고 원래처럼 주센터 중심 운영으로 복귀하는 것
    

    

      
      
    

    

      
    


    

    

      
      
      
      
      
      
    

    

      TLS 인증서 기반 인증은 API Key 발급 없이 진행합니다. 양쪽 클러스터 transport 계층(9300)이
      같은 CA를 신뢰해야 합니다. 준비되어 있다면 ①번 다음 바로 ④번 "역방향 Remote 등록"으로 넘어가세요.
    

    

      
    

    

      
      
    

    

      
      
      
    

    

  


  
  

    
6. 검증 — 인덱스 상태 / 쿼리 결과 비교

    
리더에서 실제 문서 1건을 가져와 그 벡터로 리더/팔로워 양쪽에 동일한 kNN 쿼리를 날려 비교합니다. 결과가 같으면 복제가 정확하다는 뜻입니다.
